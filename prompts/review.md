# 异构审查 Prompt

本 prompt 用于阶段 ⑥，对执行产出进行异构审查。

---

## 指令

你是审查方。你的任务是审查执行方的交付物，检查是否存在偏差、遗漏或违规。

### 读取以下文件

1. `workflow/registry.yaml` — 获取当前状态
2. 当前有效的 `spec-pack.v{N}.md`
3. 执行方的 `evidence-handoff.v{N}.md`
4. 执行方的 Understanding Lock 文件
5. 实际交付物（代码文件 / 设计规范文件）

### 审查维度

按以下四个维度逐一检查，每个维度给出结论：

#### 1. 需求漂移

- 实现/设计是否偏离了 Spec Pack 中定义的功能？
- 是否有 Spec Pack 中没有提到的功能被偷偷加入？
- 是否有 Spec Pack 中明确要求的功能被遗漏？

#### 2. 设计漂移

- 实现是否违反了设计规范中的约定？
- 设计是否超出了 Spec Pack 定义的功能边界？

#### 3. 未声明假设

- Evidence Handoff 中列出的假设是否合理？
- 是否存在执行方没有声明但实际做出的假设？
- 这些假设是否违反了锁定参数？

#### 4. 硬约束违反

- 是否违反了 Spec Pack 中的任何约束条件？
- 是否违反了锁定参数？
- 安全性、性能、兼容性等硬约束是否被满足？

### 输出格式

```markdown
## 审查报告

**审查方**：[Agent 名称]
**审查对象**：[被审查的 Agent 和交付物]
**日期**：[YYYY-MM-DD]

### 1. 需求漂移
- **结论**：[无漂移 / 轻微漂移 / 严重漂移]
- **详情**：[具体发现]

### 2. 设计漂移
- **结论**：[无漂移 / 轻微漂移 / 严重漂移]
- **详情**：[具体发现]

### 3. 未声明假设
- **结论**：[无 / 有但合理 / 有且有风险]
- **详情**：[具体发现]

### 4. 硬约束违反
- **结论**：[无违反 / 有违反]
- **详情**：[具体发现]

### 总体建议
- [ ] 建议 PASS
- [ ] 建议 REVISE（附修改建议）
- [ ] 建议 REDO（附原因）
```

### L3 双异构审查

在 L3 任务中，审查是交叉进行的：
- Codex 审查 Gemini 的设计规范（检查设计是否可实现、是否与代码一致）
- Gemini 审查 Codex 的代码实现（检查实现是否符合设计规范、交互是否完整）

两份审查报告分别保存，由 Claude 在决策阶段综合判断。
