# Spec Challenge Prompt

本 prompt 用于阶段 ② Spec Challenge。
在 L2（可选）和 L3（强制）任务中，执行 Agent 在开工前对 Spec Pack 进行预检。

---

## 指令

你即将执行一个任务，但在开始之前，你需要先审查 Spec Pack 的质量。

请读取：`workflow/artifacts/spec-pack.v{当前版本}.md`

### 如果你是 Codex（代码实现方），检查以下维度：

1. **可实现性**：规格中描述的功能，在当前技术栈下是否可以实现？有没有技术上不可能或极其昂贵的要求？
2. **可测试性**：每条验收用例是否足够具体，能写出自动化测试？是否有模糊的描述（如"性能良好""用户体验好"）？
3. **歧义与遗漏**：是否有两种以上合理解读的描述？是否有功能被提到但没有对应验收用例？
4. **锁定参数合理性**：锁定参数是否会阻碍实现？可自决参数的范围是否足够？

### 如果你是 Gemini（设计方），检查以下维度：

1. **可设计性**：规格中描述的交互，是否有足够信息来产出设计规范？
2. **状态完整性**：是否涵盖了所有关键状态（空状态、加载中、错误、成功、边界情况）？
3. **体验说明**：是否缺少关键的交互说明（动画、过渡、响应式行为）？
4. **与功能边界的一致性**：设计需求是否超出了 PRD 定义的功能范围？

### 输出格式

```markdown
## Spec Challenge 结果

### 通过项
- [列出没有问题的维度]

### 发现的问题
- **问题 1**：[描述] → **建议**：[修改建议]
- **问题 2**：[描述] → **建议**：[修改建议]

### 结论
- [ ] 规格可以按当前版本执行
- [ ] 规格需要修改后再执行（列出必须修改的项）
```

### 注意事项

- 不要给出泛泛的建议（如"建议补充更多细节"）。每个问题必须指向 Spec Pack 中的具体段落或字段
- 你的 Challenge 结果将提交给 Claude 做裁决，Claude 决定是否采纳修改
